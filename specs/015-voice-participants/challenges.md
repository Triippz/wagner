## Detailed Finding Narratives

### C5 — Missing test tasks for FR-012, FR-013, FR-014, FR-015

**What could go wrong:** FR-012 (audio stays local) has no automated test. If an implementation error ever added audio data to a hub sync payload, there is no CI gate to catch it. The only protection is Article IX in the constitution — but the constitution's verification requirement is explicitly `"a test asserts a representative run's default sync transmits no transcript or code field"` (`constitution.md:147`). This test does not exist in tasks.md.

FR-015 (toggle gate) has no test. If the toggle wiring is broken, the microphone could capture and the TTS could speak when the operator has explicitly turned voice off. A user who is on a call and turns the voice toggle off expects silence — a broken toggle is a user-visible privacy regression.

FR-014 (typed error, no panic) is covered by T035 which is a UI-surfacing implementation task, not a test task that verifies the error surface before that implementation exists.

**What would fix it:** Four new `[P]`-tagged test tasks, inserted before their implementation counterparts:

- `[P] Test: hub-sync payload contains no audio or transcript field when voice is active and a run syncs; assert with the stub hub (D-TEST-4). Place before any hub-integration impl task.`
- `[P] Test: schema-validation is called at every voice command emit site — unit test that FakeDispatch receives a schema-valid payload and that an invalid payload is rejected before emit. Place before T018 (intake impl).`
- `[P] Test: VoiceStatus surfaces a typed VoiceError variant (not a panic/unwrap) when mic permission is denied, STT sidecar is unreachable, and TTS sidecar is unreachable — three separate test cases. Place before T035.`
- `[P] Test: voice toggle off suppresses audio capture and TTS output — with voice off, scripted PTT trigger produces no AudioChunk and scripted speakable event produces no FakeTts invocation. Place before T017 (PTT impl).`

### C6 — SC-005/SC-006 are unratified assumptions, not evidence-driven targets

**What could go wrong:** The spec asserts `"p95 utterance-end → command ≤ 1.5 s"` as a success criterion. If whisper.cpp on M-series arm64 routinely takes 2.2 s for a 4-second utterance (which is plausible for a cold model), the system ships missing its own SC-006 with no CI gate to detect it. The manual `T040 performance pass` is a one-time check, not a regression guard. SC-005's `"1 false-accept/hour"` is explicitly flagged as having `"no verified benchmark"` — there is no source from which this number was derived. If the custom "Hey Wagner" model achieves 5 false-accepts/hour (a realistic outcome for a first synthetic-TTS-trained model), SC-005 fails on first use.

**What would fix it:** For SC-006, add a benchmark citation — even a rough one: `"whisper.cpp base.en model on M2 at 16 kHz: median RTF 0.05–0.1 per published benchmarks (https://github.com/ggerganov/whisper.cpp README); a 3-second utterance transcribes in ~0.15–0.3 s, leaving >1.2 s for the remainder of the pipeline."` This is a defensible Article II(c) rationale. For SC-005, explicitly state: `"This target is set by engineer judgment as a usability threshold; it will be measured during the T004/T037 build and adjusted if the model cannot meet it."` This is an Article II(c)-compliant Assumption with stated rationale. Add a headless latency regression test (using pre-recorded PCM + FakeStt with a known delay) that fires in CI.

### H1 — "All four P1" violates Article IV's independence requirement

**The specific technical problem:** tasks.md:67 explicitly states T024 `"reuse the US1 Stt→command path"`. This means implementing US2 (wake) alone — without US1 — produces a detector that fires on wake-word but has no path to submit a command. That is not a viable MVP. Article IV's independence requirement (`constitution.md:68`) is not just about testability; it requires that `"implementing only that story MUST yield a viable MVP that delivers the user the value the spec promises."` A wake detector that detects but cannot act does not deliver the value US2 promises.

**The Complexity Tracking entry does not resolve this.** plan.md:145 states: `"A strict P1→P4 ladder was offered and declined; 'both now' is the stated product intent."` This addresses why the engineer wants to ship all four together, not why US2 independently yields an MVP. These are different claims. The Complexity Tracking entry justifies the scope decision (Gate IV); it does not address the Article IV definition of P1.

### H3 — EC-002 and FR-011a are contradictory for the half-duplex case

**Scenario that reveals the contradiction:** The AEC build spike (T004) fails on macOS arm64. The system falls back to half-duplex (FR-011a fallback path). A user says "Hey Wagner, stop" while TTS is playing. Per FR-011a half-duplex: the wake detector is gated during TTS — the utterance is NOT detected. Per EC-002: `"full-duplex — AEC (FR-011a) lets the detector fire over playback; on detection the system ducks/stops the current TTS."` A developer reading EC-002 implements barge-in behavior. A developer reading FR-011a implements a TTS gate. These produce different systems. The spec needs to tell them which behavior to implement when the AEC spike fails.
